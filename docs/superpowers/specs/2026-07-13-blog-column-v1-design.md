# 小陌博客：「博客搭建手记」四篇版内容设计

日期：2026-07-13
状态：已确认，进入实施计划

## 1. 背景与判断

博客已经具备飞书发布、静态部署、分类、专栏、标签、站内搜索、专栏连续阅读和相关文章。站点功能和信息架构已足够完整，当前密度瓶颈来自内容供给：公开归档只有两篇约一分钟文章，而飞书首发文仍保留“用于验证”和“测试文章更新”等占位表达。

本阶段不再增加导航或客户端功能，而是把现有「博客搭建手记」完成为四篇可连续阅读的真实专栏。内容继续以飞书文档为原稿、多维表格为发布清单，不手工编辑飞书生成的 Markdown。

## 2. 目标

- 将「博客搭建手记」扩展为序号连续的 `01–04` 四篇专栏。
- 重写现有飞书首发文，删除占位和测试语句，保留已公开的稳定 URL。
- 新增两篇基于本仓库真实实现的技术文章，不编造功能、数据或运行结果。
- 让专栏连续阅读、相关文章、搜索和标签真正产生有用结果。
- 保留纯静态站点、现有飞书字段和当前 GitHub Actions 工作流。

## 3. 非目标

- 不新增飞书 Base 字段、工作流触发条件或仓库内容模型。
- 不增加评论、统计、账号、服务端搜索或其他运行时依赖。
- 不在本轮重做首页、文章页或分类目录视觉。现有页面密度将在四篇真实内容上重新评估。
- 不为新文章制作大体积封面。封面响应式优化和社交分享卡片作为独立后续项目。
- 不为填充数量发布无法复用的流程记录、日志或占位短文。

## 4. 专栏结构

### 4.1 第 01 篇：欢迎来到小陌的博客

- 来源：现有手写 Markdown，正文保持不变。
- Slug：`welcome`
- 作用：介绍博客定位和整体架构，作为专栏入口。
- 分类：随笔
- 专栏：博客搭建手记
- 专栏序号：1
- 精选：否，将现有 `featured: true` 改为 `false`。

本轮不改写第 01 篇正文，只调整精选元数据，避免首页同时存在两篇精选。

### 4.2 第 02 篇：从飞书到博客：一条可回滚的静态发布链路

- Slug：`published-from-feishu`（保持不变）

- 封面：保留现有封面附件。

正文至少包含以下二级标题：

1. 为什么把飞书当写作桌面

- 分类：技术

## 5. 写作规范

每篇实操文章使用同一类信息结构，但不强制机械的模板句式：

1. 开头用一至两段说明真实问题和最终结论。
2. 用二级标题组织实现过程，每个章节回答一个明确问题。
3. 涉及仓库实现时，只描述已存在的配置、安全边界、测试和工作流。
4. 不写无法从仓库、GitHub Actions 或公开站点验证的数据。
5. 结尾提供一份可直接复用的检查清单，而不使用泛化的励志性总结。
6. 中文表达保持具体、自然，避免宣传口号、三段式堆叠和 AI 式过度概括。

篇幅以“说清问题”为准，不设硬性字数；预期每篇实操文章产生至少 3 分钟阅读时长。

## 6. 飞书发布流程

### 6.1 内容准备

1. 读取现有飞书首发文文档和 Base 记录，确认稳定的文档、记录与 Slug，并保存发布前快照。
2. 为第 02 篇创建一份完整替代文档，发布时只切换同一条 Base 记录的文档链接，保持 `published-from-feishu` URL 不变；旧文档不删除。
3. 新建第 03、04 篇飞书文档，先在 Base 中以“草稿”状态创建记录。
4. 补齐十二个现有字段中的公开元数据；新文章封面留空。
5. 将手写第 01 篇的 `featured` 调整为 `false`，但不改写正文。

### 6.2 发布门禁

在任何记录切换为“已发布”前：

- 正文没有占位文字、未完成段落、内部凭据、本地路径或私有 URL。
- 每篇至少有两个有意义的二级标题，可以生成真实目录。
- 摘要与正文相符，不使用“测试”、“验证文章”或“以后补充”等占位表达。
- 分类、专栏、序号、标签和精选值与第 4 节一致。
- 专栏序号 1–4 唯一且连续，所有 Slug 唯一且稳定。

第 02 篇保持“已发布”，但在最终发布窗口前仍指向旧文档；第 03、04 篇必须保持“草稿”，直到三篇替代/新文档都通过内容检查。发布窗口内先将第 02 篇记录切换到替代文档，再通过 Base 批量更新将第 03、04 篇切换为“已发布”。任一步骤失败时，立即恢复第 02 篇的旧文档链接并把第 03、04 篇退回“草稿”。自动化可能产生重复的 workflow dispatch，但同步是幂等的，最终只会保留同一组公开内容。

### 6.3 同步与部署

1. 运行飞书同步，让生成的 Markdown、媒体和 manifest 由现有脚本产生。
2. 运行完整测试、Astro 检查和生产构建。
3. 确认飞书同步工作流推送生成内容并直接部署 Pages。
4. 验收线上搜索、专栏顺序、相关文章、RSS、sitemap 和公开 URL。

## 7. 数据边界与安全

- 仅在飞书文档和“博客文章” Base 表中编辑内容，不手工修改 `src/content/posts/feishu/`、`public/media/feishu/` 或 `.feishu-manifest.json`。
- 不在文档、Base、Git 提交或公开页面中写入 App Secret、PAT、record ID、document ID、file token 或内部 URL。
- 文章可以解释字段名和工作流概念，但不引用实际凭据、内部标识或未脱敏日志。
- 两篇新文章在“草稿”状态下不会进入仓库或搜索索引。
- 第 02 篇旧文档保留但不再由 Base 引用，回滚时可恢复原文档链接和发布前元数据。
- 同步或构建失败时，上一版 Pages 继续在线；不绕过验证直接提交半成品。

## 8. 错误处理

- 飞书文档或 Base 权限不足：停止内容修改或发布，先恢复应用可读权限。
- 新文章字段不合法：记录保持“草稿”，根据字段名修正后重试。
- 飞书文档含未支持块：转换为支持的标题、段落、列表、引用、代码或简单表格，不做静默丢弃。
- 专栏序号、Slug 或路由冲突：整次同步失败，修复元数据后重跑，不局部发布。
- 同步期间文档修订号变化：允许脚本自动重试一次；仍不稳定时终止发布。
- 构建、测试或 Pages 部署失败：保留失败日志和上一版站点，修复后重跑失败工作流。

## 9. 质量检查

### 9.1 内容检查

- 使用自然中文审阅三篇飞书文章，删除 AI 式口号、空泛总结和无依据结论。
- 逐项核对文章中描述的字段、工作流、安全限制和恢复路径是否与仓库一致。
- 检查标题层级、列表、代码与链接，避免同步转换后形成空标题或断裂段落。
- 确认第 02 篇不再出现“用于验证”、“测试文章更新”等占位内容。

### 9.2 技术检查

- 飞书 CLI 读回三条记录，确认标题、Slug、分类、专栏、序号、标签、精选和状态。
- 飞书同步后生成三篇飞书 Markdown，与手写第 01 篇组成连续 `01–04`。
- `npm run verify` 全量通过，构建产物包含三个飞书文章 URL 和专栏目录。
- 搜索索引不含飞书内部标识、凭据、私有 URL 或本地路径。
- 搜索“飞书发布”、“GitHub Actions”、“失败恢复”和“静态部署”时，目标文章出现在前三条。
- 线上专栏显示 `01 / 04`、`02 / 04`、`03 / 04`、`04 / 04`，上一节和下一节可以完整走通。
- 新文章、专栏页、RSS 和 sitemap 返回 HTTP 200，线上浏览器无控制台错误或水平溢出。

## 10. 验收标准

- 公开归档共有四篇文章，「博客搭建手记」序号 1–4 连续且唯一。
- 第 02 篇 URL 保持 `/posts/published-from-feishu/`，占位正文被完整替换。
- 第 03、04 篇只通过飞书文档和 Base 发布，不存在手工生成的并行内容。
- 三篇实操文章每篇至少有两个有意义的二级标题，且站点阅读时长至少为 3 分钟。
- 只有第 02 篇为精选；核心标签“飞书发布”、“自动化”和“静态部署”至少各覆盖两篇文章。
- 从首页、搜索或专栏页到任一目标文章不超过两次点击。
- 飞书同步、站点验证、GitHub Pages 部署和线上实网验收全部成功。

## 11. 后续顺序

四篇版发布后，先用真实内容重新测量首页和文章页的首屏密度。下一个独立小项目优先级为：

1. 响应式封面与图片体积预算。
2. 文章社交分享卡片与更完整的 Article 结构化数据。
3. 首页首屏、移动文章头部和归档对齐的精准密度调整。
4. 将少量高风险 UI 源码正则门禁迁移为真实浏览器烟雾测试。
