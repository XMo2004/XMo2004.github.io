# 小陌个人博客与飞书发布链路设计

日期：2026-07-12
状态：已确认，进入实施

## 1. 目标

上线一个公开访问的个人博客，首发地址为 `https://xmo2004.github.io/`。博客由 GitHub Pages 静态托管，不运行长期在线的应用服务器。作者在飞书文档中完成写作，在飞书多维表格中维护发布信息；将状态改为“已发布”或点击发布按钮后，飞书自动化立即触发 GitHub，同步文章并重新部署网站。

上线验收同时要求：

- 首页、文章列表、文章详情、标签页、关于页、RSS、站点地图和 404 页面可用。
- 电脑与手机均可正常阅读，深浅色主题可切换。
- 发布、修改、下线三种内容变更均能反映到静态网站。
- 飞书凭据和 GitHub 触发令牌不进入代码仓库。
- 飞书同步失败时流水线明确失败并保留错误日志，不静默发布残缺文章。

## 2. 已采用方案

### 2.1 技术栈

- 站点生成：Astro 7，使用静态输出和内容集合。
- 样式：原生 CSS 与少量无依赖浏览器脚本，不引入重量级 UI 框架。
- 内容中间层：仓库内 Markdown 与本地化图片，保留 Git 历史以便审计、回滚和迁移。
- 自动化：GitHub Actions。
- 托管：GitHub Pages，仓库为公开的 `XMo2004/XMo2004.github.io`。
- 飞书：新版文档 `docx` 写正文，多维表格作为发布控制台，自建应用只读拉取内容。

### 2.2 方案比较

1. **多维表格显式发布 → GitHub Actions → GitHub Pages（采用）**
   - 优点：发布语义明确；不需要服务器；现有 GitHub 登录可直接部署；内容有 Git 快照；维护成本最低。
   - 代价：首次需要创建飞书自建应用、多维表格和一条自动化。
2. **飞书文档编辑事件 → Cloudflare Worker → 静态托管**
   - 优点：可以监听每次文档变化。
   - 代价：编辑不等于发布；需要处理 challenge、验签、重试和去重，并增加一个 Serverless 运行面。
3. **GitHub Actions 定时轮询飞书**
   - 优点：配置最少，不需要飞书发出 HTTP 请求。
   - 代价：不是即时推送。仅保留为自动化失效时的兜底，每 30 分钟检查一次。

## 3. 产品与视觉设计

### 3.1 品牌默认值

- 站点名：小陌的博客
- 英文标记：XMO / NOTES
- 简介：关于技术、成长与日常的长期笔记。
- 作者：小陌

这些值集中放在站点配置文件中，后续更名不影响页面结构或飞书内容。

### 3.2 视觉系统

整体采用安静的编辑式版面，而不是常见的彩色渐变模板：

- 浅色背景为暖纸白，正文为墨黑；深色背景为炭黑，正文为暖灰白。
- 强调色使用陶土橙，辅助色使用鼠尾草绿。
- 标题使用可靠的中文衬线系统字体，界面与正文使用系统无衬线字体，避免依赖在中国大陆不稳定的在线字体服务。
- 卡片以留白、细边线和排版层级区分，不使用装饰性图标堆叠。
- 动效仅用于页面入场、链接下划线和主题切换，并尊重 `prefers-reduced-motion`。

### 3.3 页面结构

- 首页：作者简介、精选文章、最新文章、主题入口与 RSS 订阅。
- 文章列表：按发布日期倒序，支持标签筛选。
- 文章详情：标题、摘要、日期、阅读时长、标签、正文、桌面端目录、上一篇/下一篇。
- 标签页：所有标签与对应文章。
- 关于页：简洁的作者说明与站点工作方式。
- 404：保留同一视觉语言并提供返回入口。

不在首版加入评论、账户、管理后台、数据库、全文搜索和访问统计；飞书已经承担内容后台职责。

## 4. 飞书内容模型

### 4.1 多维表格字段

数据表名为“博客文章”，每行对应一篇文章：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| 标题 | 文本/索引列 | 必填；为空时可回退为飞书文档标题 |
| 文档链接 | 超链接 | 必填；必须指向新版飞书文档 |
| Slug | 文本 | 必填；稳定的 ASCII 小写路径，如 `hello-world` |
| 摘要 | 多行文本 | 必填；用于列表与 SEO 描述 |
| 标签 | 多选 | 可选 |
| 封面 | 附件 | 可选；为空时使用排版式封面 |
| 发布日期 | 日期 | 必填 |
| 状态 | 单选 | `草稿`、`已发布`、`已下线` |
| 精选 | 复选框 | 可选 |

流水线只读取状态为“已发布”的记录。状态改回“草稿”或“已下线”后，下一次同步会从公开站点中移除对应文章。同步时每次读取全部已发布记录，因此重复触发是幂等的，也不会依赖某次 webhook 中是否成功携带记录字段。

### 4.2 飞书文档支持范围

首版转换器支持：

- 普通段落与 1–6 级标题
- 粗体、斜体、删除线、行内代码、链接
- 有序列表、无序列表、待办、引用、分隔线
- 代码块与语言标识
- 图片下载并改写为本地地址
- 简单表格

分栏、思维导图、画板、嵌入多维表格、第三方小组件等复杂块不做不完整降级；检测到时同步直接失败并指出块类型，作者可先将其改成支持的内容。

### 4.3 图片与一致性

- 文档图片和多维表格封面通过飞书素材下载 API 拉取，按内容哈希命名，保存到 `public/media/feishu/`。
- 发布前后各读取一次文档 `revision_id`。若正文在同步期间发生变化，则重试一次；仍变化时终止本次同步，避免混合两个版本。
- 生成内容写入独立的 `src/content/posts/feishu/`，不会删除仓库内手写的示例或欢迎文章。
- 清单文件记录 `record_id`、`document_id`、`revision_id`、slug 和资源哈希，用于增量判断与清理已下线资源。

## 5. 发布数据流

1. 作者在飞书文档中完成正文。
2. 作者在“博客文章”表中填写元数据，将状态设为“已发布”或点击发布按钮。
3. 飞书自动化使用 HTTPS `POST` 调用：
   `https://api.github.com/repos/XMo2004/XMo2004.github.io/actions/workflows/sync-feishu.yml/dispatches`
4. 请求头包含 GitHub 推荐的 API 版本、JSON 类型，以及仅限该仓库且仅有 Actions 写权限的细粒度令牌；请求体固定为 `{"ref":"main"}`。
5. `sync-feishu.yml` 获取飞书 `tenant_access_token`，读取多维表格和文档，将已发布记录完整同步为 Markdown 与本地资源。
6. 有内容变化时，Action 以机器人身份提交到 `main`；无变化则正常结束。
7. `deploy.yml` 在 `main` 更新时运行测试、Astro 检查和静态构建，并通过 GitHub Pages Actions 发布构建产物。
8. 兜底定时任务每 30 分钟运行一次同步；仓库也保留手动触发入口。

## 6. 安全与权限

- GitHub 触发令牌：细粒度 PAT，仅允许访问 `XMo2004.github.io`，仅授予 Actions 读写权限，存放在飞书自动化的 Authorization 请求头中。飞书运行日志的“流程设置”会显示完整请求头，因此只有博客所有者可以管理自动化，令牌使用短有效期并定期轮换。
- 飞书 App ID、App Secret、Base App Token、Table ID：保存为 GitHub Actions repository secrets，不进入仓库或构建产物。
- 飞书应用只申请读取新版文档、下载文档素材、只读多维表格所需权限；应用还必须被添加为目标文档和多维表格的协作者。
- 同步日志不得输出 App Secret、访问令牌或完整 Authorization 头。
- GitHub Pages 部署使用仓库自带的 `GITHUB_TOKEN`，权限限定为 `contents: read`、`pages: write`、`id-token: write`。

## 7. 错误处理

- 缺少凭据：同步任务在网络请求前失败并列出缺失的环境变量名。
- 飞书限流：对 429 和官方限流错误实施带抖动的指数退避，调用速率保持低于每秒 5 次。
- 字段错误：同一次同步汇总所有无效记录，列出记录 ID 与错误，不发布部分成功结果。
- Slug 冲突或非法路径：同步失败，防止覆盖其他文章或目录穿越。
- 文档不可访问：返回明确的 document ID 和共享权限提示。
- 未支持块：记录块 ID 与类型并失败，避免静默丢失正文。
- 部署失败：上一版 GitHub Pages 继续在线；失败日志保留在 Actions 中，可手动重试。

## 8. 测试与验证

采用测试优先方式实现：

- 单元测试：URL/document ID 解析、富文本转义、Block 到 Markdown、slug 校验、元数据映射、清单差异。
- 契约测试：以脱敏飞书 API fixture 验证分页、限流重试、图片下载和错误聚合。
- 构建测试：运行 Astro 类型检查与生产构建，验证 RSS、sitemap、文章 URL 和 404 输出。
- 浏览器检查：在桌面和手机尺寸检查首页、文章页、主题切换、键盘焦点、控制台错误、文本溢出和响应式布局。
- 线上检查：访问正式 Pages URL，确认 HTTPS、首页、示例文章、RSS 与 sitemap 均返回 200。
- 端到端飞书验收：发布一篇专用测试文档，确认自动化触发、同步提交、Pages 部署与线上正文一致；随后下线并确认页面消失。

## 9. 上线边界

本轮上线使用 GitHub 提供的 `xmo2004.github.io` 域名，因为尚未提供自定义域名。项目会保留标准的 `site` 配置与自定义域名说明，未来绑定域名无需重构。只有当正式 Pages 站点可访问、飞书推送链路完成真实发布与下线测试、且使用说明写清楚后，目标才算完成。
