# 小陌博客：响应式封面与图片体积门禁设计

日期：2026-07-14
状态：已确认，待实施计划

## 1. 背景

博客当前只有一张真实文章封面。它是 `1672 × 941`、约 `2.0 MiB` 的无透明 PNG。首页精选卡片、文章归档和分类、专栏、标签列表都直接引用同一原图；手机、缩略图和高分辨率桌面没有区分资源尺寸。

对当前封面进行不落盘试算后，使用相同画面生成 `320 / 640 / 960 / 1440` 四档 WebP，质量参数为 `78`，文件分别约为 `4 / 10 / 19 / 39 KiB`，合计约 `72 KiB`。因此可以在不改变视觉设计的前提下显著降低下载量，并把同一规则自动应用到以后从飞书发布的封面。

## 2. 目标

- 保持现有封面裁切、卡片比例、色彩和交互不变。
- 飞书同步时自动生成响应式封面，不要求作者手工压图或维护多个附件字段。
- 手机、首页精选卡片和文章列表缩略图按实际显示宽度选择合适资源。
- 为封面源文件、解码像素和生成结果设置明确上限，避免大图拖慢同步、仓库和页面。
- 维持现有同步的确定性、幂等性、原子替换和失败回滚。
- 完成当前封面的自动迁移，删除公开目录中的 2 MiB 原始 PNG，只保留优化后的派生资源。

## 3. 非目标

- 不改变封面内容，不重新生成图片，不修改飞书多维表格字段。
- 不把封面添加到文章正文头部；本轮只优化已有卡片和列表中的封面。
- 不处理正文图片的响应式转换；正文媒体继续沿用现有下载、大小限制和内容寻址规则。
- 不增加图片 CDN、服务端图片接口或客户端图片处理脚本。
- 不在本轮制作 Open Graph 分享图。

## 4. 方案选择

### 4.1 采用：飞书同步时生成响应式 WebP

同步脚本下载封面源文件后，在内存中完成校验、自动方向修正、缩放和 WebP 编码。派生文件继续使用 SHA-256 内容寻址，并与 Markdown、manifest 一起进入现有暂存目录和原子替换流程。

优点：每个未来封面自动获得相同优化；构建只消费已经准备好的静态资源；同步无变化时不会产生新提交；失败不会污染上一版生成树。

### 4.2 不采用：Astro 构建时重复转换

该方案可以少改同步代码，但每次部署都要重新编码，生成结果不进入当前 manifest，内容元数据也难以在构建前获得可靠尺寸。它会把发布数据准备和页面构建混在一起，不符合现有边界。

### 4.3 不采用：只手工压缩当前封面

单张替换实现最简单，但下一次从飞书上传大图时问题会复现，也无法让缩略图与首页卡片选择不同尺寸。

## 5. 处理流程

### 5.1 输入门禁

封面沿用现有可信飞书媒体下载，不改变正文图片规则。封面专属门禁为：

- 源文件最多 `5 MiB`。
- 解码后最多 `24,000,000` 像素。
- 必须能被 Sharp 解码为有正整数宽高的静态位图。
- 多页或动画图片不作为封面发布；作者需上传静态版本。
- 自动应用 EXIF 方向，不放大尺寸不足的图片。

这些门禁只针对封面。正文单文件上限仍为现有 `10 MiB`。

### 5.2 派生规格

- 目标宽度：`320`、`640`、`960`、`1440` 像素。
- 当源图窄于某一目标宽度时跳过该档，并把源图实际宽度作为最后一档；不生成重复尺寸。
- 输出格式：WebP。
- 编码：`quality: 78`、`effort: 4`、`smartSubsample: true`。
- 所有派生文件合计不得超过 `1 MiB`。
- 最大公开宽度为 `1440`；下载到内存的源文件不直接写入公开目录，除非同一素材也被正文引用。

Sharp 作为直接、锁定版本的生产依赖，而不是依赖 Astro 的间接安装。

### 5.3 内容元数据

飞书生成文章的 `cover` 从字符串升级为以下结构：

```yaml
cover:
  src: /media/feishu/<largest-hash>.webp
  width: 1440
  height: 810
  variants:
    - src: /media/feishu/<320-hash>.webp
      width: 320
    - src: /media/feishu/<640-hash>.webp
      width: 640
    - src: /media/feishu/<960-hash>.webp
      width: 960
    - src: /media/feishu/<1440-hash>.webp
      width: 1440
```

内容 schema 暂时兼容既有字符串封面，保证代码先部署、内容后同步的过渡阶段不会构建失败。结构化对象必须满足：路径为站内绝对路径，尺寸为正整数，variants 至少一项、按宽度严格递增、最后一项与 `src / width` 一致，所有路径唯一。

manifest 继续使用版本 `2`，只记录实际写入公开目录的派生资源。它不包含飞书文件 token、记录 ID、文档 ID 或源附件 URL。

## 6. 页面渲染

新增一个只负责装配响应式属性的封面组件，`PostCard` 和 `PostRow` 继续负责各自布局：

- 结构化封面输出 `src`、`srcset`、`sizes`、`width`、`height`、空 `alt` 和异步解码。
- 旧字符串封面继续输出普通 `<img>`，用于安全迁移和手写内容兼容。
- 首页精选卡片使用 `loading="eager"` 与 `fetchpriority="high"`，因为它是首屏最大视觉资源。
- 文章列表继续使用 `loading="lazy"`。
- 精选卡片的 `sizes` 为 `(max-width: 48rem) calc(100vw - 2rem), 30rem`。
- 列表缩略图的 `sizes` 为 `(max-width: 30rem) 1px, (max-width: 48rem) 5.25rem, 7rem`。极窄屏下缩略图仍由现有 CSS 隐藏；即使浏览器提前选择资源，也只会选择最小档。
- 图片保持装饰性空替代文本，文章标题仍是可访问名称。

不改变现有 `object-fit: cover`、卡片尺寸、边框、悬停缩放或移动端隐藏规则。

## 7. 同步边界与错误处理

封面处理发生在 `buildNextState` 内、`writeStage` 之前：

1. 读取稳定文档和发布记录。
2. 下载并计入现有网络媒体预算。
3. 校验封面输入并生成所有派生图。
4. 将派生图加入文章资产、全局资产和 manifest。
5. 只有全部文章、正文媒体和封面都成功后才写暂存树并执行原子替换。

任一封面无法解码、超过输入/像素/输出预算或为动画时，整次同步在构建阶段失败。已有 Markdown、媒体、manifest 和线上 Pages 不变；错误只暴露公开阶段与文章 slug，不输出飞书内部标识。

## 8. 迁移与发布顺序

1. 先发布兼容字符串和结构化对象的代码版本。
2. 在 GitHub Actions 中运行完整验证并部署，确认旧 2 MiB PNG 仍可正常显示。
3. 手动触发现有飞书同步工作流，让真实封面通过新处理器重新生成。
4. 同步提交应删除旧 PNG，新增四档内容寻址 WebP，并把文章 frontmatter 改为结构化封面。
5. 同步工作流再次执行完整验证并部署 Pages。

如果第 3–5 步失败，保留第 2 步已上线的兼容版本和旧封面，不手工编辑生成目录绕过同步。

## 9. 测试策略

### 9.1 处理器测试

- 先写失败测试，再实现封面处理器。
- 对确定输入生成严格递增的目标宽度、正确高度和 WebP 内容类型。
- 相同输入与参数生成相同内容寻址路径。
- 小图不放大且不产生重复档位。
- EXIF 方向后的宽高正确。
- 源文件超过 `5 MiB`、超过 `24 MP`、动画、无法解码或派生总量超过 `1 MiB`时安全失败。

### 9.2 同步与内容测试

- 更新同步夹具为真实可解码图片。
- 断言 frontmatter 使用结构化封面，公开目录和 manifest 只包含真实派生资源。
- 断言第二次相同同步不替换文件系统。
- 现有校验失败、替换失败、事务日志恢复和旧树字节不变测试继续通过。
- 内容 schema 同时接受旧字符串与合法对象，并拒绝乱序、重复或不一致的 variants。

### 9.3 页面与线上验收

- 构建产物包含正确的 `srcset`、`sizes`、固有尺寸、加载优先级和懒加载属性。
- 320、768、1440 三档无破图、布局位移、裁切变化或水平溢出。
- 浏览器 `currentSrc`：手机精选卡片选择 `640w` 或更小；桌面精选卡片不超过 `960w`；列表缩略图选择 `320w`。
- 当前封面四档派生文件总量小于 `128 KiB`，旧 2 MiB PNG 不再出现在公开目录、manifest 或 HTML 中。
- 站点完整测试、Astro 检查、生产构建、首次代码部署、飞书同步部署均成功。

## 10. 验收标准

- 视觉效果与当前线上一致，封面在所有卡片中正常显示。
- 飞书只需一份封面附件，后续同步自动生成响应式资源。
- 当前公开封面从单个约 `2.0 MiB` PNG 迁移为最多四个合计小于 `128 KiB` 的 WebP。
- 页面根据显示宽度选择资源，不再让手机和缩略图下载最大封面。
- 超预算或不可处理的封面无法进入生成树，上一版站点保持可用。
- 同步仍然确定、幂等、无内部标识泄漏，并可由现有事务恢复机制安全回滚。
