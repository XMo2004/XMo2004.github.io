## 为什么把飞书当写作桌面

我不想在后台表单里写长文，也不想为了改一个标点就打开代码编辑器。飞书文档很适合留在我已经熟悉的写作节奏里：先起草，再调整标题和段落，图片也放在正文旁边。多维表格则只做发布清单，保存标题、Slug、分类、专栏序号和状态等结构化信息。正文与发布决定分开后，写作时少了不必要的干扰。

这个分工还有一个实际好处：文档能不能被读取，与这篇文章该不该上线，是两件事。文档负责内容，记录只有进入“已发布”状态才会进入同步输入。草稿可以反复改，不会因为文档已经共享给应用就意外出现在公开站点。

## 从“已发布”到静态页面的数据流

同步客户端向多维表格发起查询时，筛选条件就是状态完全等于“已发布”。返回的记录不会马上写进站点。脚本先检查 Slug 格式、日期、分类、精选布尔值和封面形状；专栏与专栏序号必须同时有值或同时留空。它还会拒绝重复 Slug、手写文章的路由冲突，以及同一专栏内重复的序号。发布清单在这里是输入门，不是绕过检查的快捷入口。

记录全部通过后，脚本才读文档。它确认一次读取前后的修订号一致，再把正文转为 Markdown；图片按内容哈希命名，文章、素材和 manifest 一起组成完整的下一状态。新状态先写到临时目录，与当前生成结果逐字节比较，真的有差异才进入文件替换。修订重试和替换失败的细节留给下一篇展开，这里先记住顺序：先得到可检查的完整结果，再碰正式生成目录。

GitHub Actions 中的同步 workflow 随后运行完整站点验证，并上传这次验证得到的 Pages 产物。验证通过后，工作流才会暂存飞书生成目录、媒体目录和 manifest。若暂存区没有差异，它不创建空提交，也不进行这次静态部署；有差异时才推送生成内容，并由同一次运行部署刚才检查过的产物。

## 发布语义、幂等与稳定 URL

这条链路追求的是可重跑，不是声称每个事件只会送达一次。飞书自动化可能与定时任务前后触发，手动操作也会增加一次调度。幂等靠的是输出规则：记录按稳定顺序处理，文章和 manifest 由当前公开数据确定，素材按内容定址。相同输入产生相同文件，重复调度最终会走到“无变更”分支。

公开地址由 Base 里的 Slug 决定，而不是由源文档的内部标识决定。因此，我可以把第 02 篇换成重写后的源文档，同时保留 `published-from-feishu`；生成文件名和站点路由都不会跟着换。仓库里有一条很实在的测试：只改变内部记录和文档标识时，生成树应保持不变。对读者来说，收藏过的 URL 仍然可用。

## 我如何验证这条链路

我更在意失败时留下什么。同步测试会先生成一版完整内容，然后注入非法 Slug、持续变动的文档或文件替换失败。断言对象不只是“有错误”：字段问题必须发生在文档读取和替换之前；修订连续变化要停止本次运行；替换中途失败则执行补偿回滚，并在补偿成功后把生成文章、素材和 manifest 恢复到失败前的字节状态。

工作流端还有第二层门禁。同步完先执行完整验证，验证成功才提交生成内容，部署作业又只在确认有内容变化时启动。因此，即使 GitHub 托管运行器里的工作目录失败了，那份未完成结果也不会被提交或部署；下一次运行会从仓库中已经提交的状态重新检出。它没有把上游 API 变成一个大事务，但把公开站点的切换放到了验证之后。

## 可复用的发布清单

- 确认记录只有在准备公开时才切换为“已发布”，其余状态不进入同步结果。

- 检查 Slug、分类、日期与专栏序号；路由或序号冲突先在发布清单里修正。

- 确保文档可由同步应用读取，准备发布时暂时停下连续编辑。

- 查看 GitHub Actions 的站点验证和生成内容差异，不手工改生成目录。

- 发布后打开稳定 Slug 对应的页面；需要回滚时，恢复上一份可读源文档或元数据，再运行飞书发布流程。
