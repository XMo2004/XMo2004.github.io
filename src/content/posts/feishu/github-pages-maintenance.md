## 一次完整部署到底发生了什么

这个博客有两条会到达 GitHub Pages 的路，排查时先分清自己看的是哪个 workflow。普通部署在 `main` 分支收到 push 时触发，也支持手动运行。它固定检出 `main`，安装 Node\.js 24 和锁定依赖，执行完整站点验证，然后配置 Pages、上传生产构建产物。只有 build 作业成功，deploy 作业才会把该产物发到 GitHub Pages 环境。

飞书同步是另一条路。它由每半小时定时任务或手动调度启动，非 `main` 分支的手动运行会在读取内容之前被拒绝。同步作业生成 Markdown、媒体和 manifest，执行与普通部署相同的完整验证，提前上传已验证的 Pages 产物，再检查三个生成路径是否有差异。有变更才提交并推送，后续部署使用同一次运行里的产物。这个提交由工作流自带的凭据推送，不会再触发普通 push 部署，所以同步 workflow 自己带着 deploy 作业。

## 上线后应该验证哪些事

我会从 Actions 运行记录往下查，而不只是刷新首页。普通 push 应该出现 `Deploy site`；飞书改稿则应该出现 `Sync Feishu content`。先确认运行来自 `main`，再看依赖安装和验证步骤。如果验证失败，部署作业就不应继续。

验证通过后，再看 Pages 产物是否上传、部署步骤是否返回公开页面地址。我会打开本次变更的文章路由，而不是只看首页缓存。对飞书同步，还要核对生成内容提交确实存在变化；如果日志说内容已是最新，后面本来就不会有部署作业。

两条工作流的部署使用同一个 `pages` 并发组，并允许取消正在运行的旧任务。连续 push，或普通部署与同步部署靠得太近时，较新的运行可能取消较早的部署。看到 cancelled 先查同组是否已有更新运行成功，不要立刻把它当成构建错误；真正应上线的是当前 `main` 对应的最新成功部署。

## 为什么无内容变更时不重复部署

“无变更”分支只属于飞书同步工作流，不能类推成普通 push 都会跳过静态部署。从本地 Git 或其他外部身份推到 `main` 的提交会触发普通 workflow；同步 workflow 使用内建令牌推送生成提交，不会再次触发普通部署。普通 workflow 被触发后，快速连续的 push 可能因 `cancel-in-progress` 让较早运行被取消。飞书同步则在验证后只暂存生成文章、媒体和 manifest，再用暂存差异决定 `changed` 输出。暂存区为空时，作业记录 `changed=false` 并正常结束；它不创建空提交，部署条件也不成立。

这个判断对定时任务很重要。即使没有人改文章，同步 workflow 仍会按计划读取发布清单并验证站点；确定生成状态相同后，就不会制造一次新的内容提交和静态部署。定时运行仍能补上偶尔漏掉的触发。

## 从失败日志到恢复上一版

恢复的第一步是找到最早失败的步骤，后面的 skipped 往往只是依赖关系的结果。普通部署如果在 `Verify site` 失败，就修正仓库源码或内容并推送新提交。飞书同步如果在字段、文档或站点验证阶段失败，就修正源文档或发布元数据，再重跑 GitHub Actions。新的 deploy 成功之前，上一版已成功部署的站点仍在服务。

还有一种更容易误判的情况：生成内容已经提交到 `main`，同一次同步的 Pages deploy 却失败或被并发策略取消。此时重新跑完整同步很可能得到“无变更”，部署作业仍会跳过。先查看是否已有更新的 `pages` 运行成功；如果没有，而且原运行上传的 Pages 产物仍可用，可以重试原运行中失败的部署作业。当前 workflow 固定使用的 `upload-pages-artifact` action 没有设置 `retention-days`，该 action 默认只保留 1 天。产物已经过期、被删除或不可用时，不要只重跑部署作业；手动运行普通 `deploy.yml`，从当前 `main` 重新验证、构建和部署。即使产物仍在，若 `main` 已继续前进，也应走普通部署，避免上线旧构建。

如果上线的内容本身有误，先修正源数据，再用反向提交恢复仓库上一版，或让同步生成正确状态；不要强制改写历史。失败日志负责告诉我停在哪一步，恢复操作则要让仓库状态、构建产物和线上页面重新对齐。

## 每次发布都可以复用的维护清单

- 确认触发来源和分支：普通 push 查 `Deploy site`，飞书发布查 `Sync Feishu content`，手动运行使用 `main`。

- 找到日志里第一个失败步骤；验证通过后，再检查 Pages 产物与部署结果。

- 同步显示无变更时，不强制重发；先判断是否只是正常的定时运行。

- 部署被取消时，查同一并发组里是否有更新的成功运行。

- 生成提交成功但部署失败时，先确认原 Pages 产物仍可用，再重试原部署作业；产物过期或当前 `main` 已变化，就手动运行普通 `deploy.yml` 完成恢复。
